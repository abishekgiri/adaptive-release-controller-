# Problem Formulation: Cost-Sensitive Delayed Contextual Bandits for Continuous Deployment

## Bandit Tuple

We model continuous deployment as a **contextual bandit** with delayed, cost-asymmetric, non-stationary feedback:

```
(X, A, R, π, T)
```

| Symbol | Definition |
|--------|-----------|
| `X`    | Context space — pre-action observable features at decision time |
| `A`    | Action space — `{deploy, canary, block}` |
| `R`    | Reward (negative cost) — observed `k` steps after the action |
| `π`    | Policy — a mapping `π : X → Δ(A)` (distribution over actions given context) |
| `T`    | Trajectory — a sequence of `(x_t, a_t, r_{t+k_t})` triples |

---

## Concrete Definitions

### Context `x ∈ X`

`x` contains **only pre-action observables** — features available before the deployment decision is made.  
No outcome, risk score, or post-deploy signal may appear in `x`.

| Feature group | Examples |
|---------------|---------|
| Commit metadata | author, commit hour, files changed, lines added/deleted |
| CI signal | tests passed, build duration, lint errors, coverage delta |
| History | deploy frequency (7-day), recent failure rate, time since last deploy |
| Change type | has dependency change, touches risky path prefix |

The context is extracted by `features/extractor.py → extract_context(...)` and returned as a frozen `Context` dataclass.  
The numeric encoding for policies is `feature_vector(context) → np.ndarray`.

### Action space `A`

| Action | Semantics |
|--------|-----------|
| `deploy` | Immediate full rollout |
| `canary` | Partial rollout (e.g. 5–10% of traffic); outcome observed after canary window |
| `block`  | No rollout; change held for re-evaluation |

### Reward `r` and Cost Matrix

Reward is **negative cost**. Cost is defined by the `(action, outcome)` pair.  
The cost matrix below is the default; all values are configurable via `CostConfig`.

| Action  | Outcome         | Default cost |
|---------|-----------------|-------------|
| deploy  | failure         | 10          |
| canary  | failure         | 4           |
| block   | would succeed   | 2           |
| deploy  | success         | 0           |
| canary  | success→promote | 1           |
| block   | would fail      | 0           |

Cost asymmetry is deliberate: a missed failure (`deploy` → failure) costs far more than a missed safe change (`block` → would succeed). The `canary` action exists precisely to trade deployment speed for reduced failure cost.

## Regret

Let `π*` denote the **oracle policy** — the policy that observes the hidden state and always selects the minimum-cost action.

**Cumulative regret** over `T` decisions:

```
Regret(T) = Σ_{t=1}^{T} [ cost(a_t, o_{t+k_t}) - cost(π*(x_t), o_{t+k_t}) ]
```

where `o_{t+k_t}` is the deployment outcome observed `k_t` steps after action `a_t`.

**Best-in-hindsight regret** (diagnostic):

```
Regret_bih(T) = Σ_{t=1}^{T} cost(a_t, o_{t+k_t}) - min_{a ∈ A} Σ_{t=1}^{T} cost(a, o_{t+k_t})
```

Both quantities are reported with bootstrap 95% CIs over ≥ 30 independent seeds.

---

## Delayed Feedback

Reward for action `a_t` is **not observed at time `t`**.  
The delay `k_t` is drawn from a configurable distribution (default: `k_t ~ Geom(p)`, `p = 0.3`).  
Until the reward arrives, it is **censored**: the policy does not know the outcome.

All policy update paths route through `delayed/buffer.py`, which holds pending `(action_id, expected_observation_time)` pairs and releases rewards when the simulated clock advances past `t + k_t`.

Policies that use reward at decision time are using oracle information. That is invalid.

---

## Non-Stationarity Assumption

The joint distribution `P(o | x)` — the mapping from context to outcome — is **non-stationary**.  
Distribution shift arises from: software architecture changes, team composition changes, pipeline tooling upgrades, seasonal traffic patterns.

We model drift as a piecewise-stationary process with **abrupt shifts** between segments (primary) and optionally **gradual drift** (secondary, stress test only).

Drift is detected online via `drift/detectors.py` (ADWIN or Page-Hinkley on the reward stream).  
Upon detection, `drift/adapt.py` triggers policy reset or windowed retraining.

## Threats to Validity

| Threat | Mitigation |
|--------|-----------|
| **Circular validity**: context features also determine outcomes in the simulator | Enforced by construction: `HiddenState` and `Context` are separate frozen dataclasses with disjoint fields; `tests/test_environment.py` asserts no shared attributes |
| **Evaluation data generated by the same risk function being evaluated** | `experiments/generate_dataset.py` is retired from evaluation pipelines; all evaluation uses `environment/synthetic.py` or real CI/CD replay |
| **Offline evaluation bias**: logging policy != evaluation policy | Offline evaluation uses IPS and doubly-robust estimation in `evaluation/replay_eval.py`; IPS weights are bounded (clipping at 20×) |
| **Overfitting to synthetic drift schedule** | Drift schedule is held out from the bandit; policy has no access to drift segment labels |
| **Dataset coverage and representativeness** | Dataset choice (TravisTorrent / Apachejit) is documented with version hash in `data/README.md`; generalization claims are limited to the dataset's project mix |
| **Sensitivity to cost matrix** | `experiments/run_cost_sweep.py` sweeps cost ratios; conclusions must hold across a reasonable range |
| **Statistical multiplicity** | Pairwise comparisons use paired bootstrap with Holm–Bonferroni correction; all seeds fixed and published |
