# Project Summary — Continuous Deployment as Cost-Sensitive Decision-Making

**The problem in one sentence.** Every time an automated build passes, a system must decide whether to ship it to users, release it cautiously to a small fraction first (a *canary*), or block it — and today's tooling treats that as a prediction problem ("will this build fail?") rather than a cost problem ("what does each wrong decision actually cost?").

**The approach.** This project reframes the deploy/canary/block decision as a *contextual bandit* — a type of learning algorithm that repeatedly chooses an action, sees the outcome, and improves over time, while balancing trying new options against using what it already knows. Each decision is made from a *context* (commit size, test signals, change type, author history). The reward is the negative of an *asymmetric cost matrix*: shipping a bad change is far more expensive than blocking a safe one, and those costs are stated explicitly instead of being hidden inside a threshold. Because a deployment's real outcome is only known later, a *pending-reward buffer* holds each decision until its outcome matures, so the policy never learns from information it would not have had at decision time. Two well-known algorithms are compared — LinUCB and Thompson Sampling — against fixed rule-based baselines.

**What it found (honestly).** The results are characterized as an *operating envelope* — the conditions under which adaptive policies help, and where they do not. In a synthetic environment, when the cost of a failed deployment is much larger than the cost of an unnecessary block, the adaptive bandit beats a static rule by a widening margin (about 19–27% in the tested high-asymmetry settings, growing monotonically as the cost ratio increases). But on a small real-data sample (two public GitHub projects with a low 5.3% failure rate and little distinguishing signal), the bandit *over-blocks* and costs about 3.8% more than a simple static rule — a negative result. Among the learning policies, Thompson Sampling beat LinUCB by 6.8% on that real data but was statistically tied with the static rule. Drift-detection, meant to handle changing conditions, was net-negative at its default setting because it raised too many false alarms. The headline message is not "adaptive always wins" but "here is precisely when it does."

**Why it matters.** Deployment decisions are made constantly across the software industry, and the tools that make them are largely blind to the asymmetric cost of being wrong. This work makes that cost trade-off explicit, treats the cautious canary release as a first-class option rather than an afterthought, and maps out the specific regimes where adaptive control earns its complexity — and where a static rule is the honest choice. The findings are bounded to the synthetic environment and the small real-data sample used here; broader generalization is left to future work.
